A3
Génération correction par correction

Pour une SOLUTION_APPLICATION :

prompt · application
Génère la correction de l'Application N du chXX.
Applique le pattern patterns/[NOM-PATTERN].md.
Applique domaine-[NOM] + correction-commune + karpathy-guidelines.
Règles Karpathy :
  - Touche UNIQUEMENT le bloc %% SOLUTION_APPLICATION Application N
  - Lis les 2 blocs adjacents mais ne les modifie jamais
  - Minimum de code : connecteurs + justifications obligatoires + conclusion
  - Zéro \textbf{} décoratif non prévu par le skill
Affiche la checklist ✓ avant d'insérer. Montre le résultat. Attends ma validation.

Pour une SOLUTION_ACTIVITE :

prompt · activité
Génère la correction de l'Activité N du chXX.
Applique domaine-[NOM] + correction-commune + karpathy-guidelines.
Règles spécifiques SOLUTION_ACTIVITE :
  - Pont explicite avec acquis du collège (obligatoire)
  - Phrase ▶ \blacktriangleright après chaque notion institutionnalisée
  - Vocabulaire accessible TCS (début lycée)
  - Ne pas écrire "Conclusion"
Règles Karpathy :
  - Touche UNIQUEMENT le bloc %% SOLUTION_ACTIVITE Activité N
  - Blocs adjacents intouchables
  - Minimum de code, pas de paraphrase inutile
Affiche la checklist ✓ avant d'insérer. Attends ma validation.

B2
Vérification post-modification
prompt · vérification
Relis source.tex section [N].
Affiche toutes les \duree{} et les "% durée estimée : X min" de cette section.
Vérifie que la somme = \duree{X min} (arrondie au multiple de 5 supérieur).
📐 Règle de cohérence
Catégorie A (affichée) → \duree{X min} multiple de 5 : Section, Activite, Exemple, Application, Exercice

Catégorie B (commentaire) → % durée estimée : X min, entier libre : Definition, Propriete, Theoreme, Remarque, Techniques, Preuve

Formule : \duree{} section = Σ(A + B) arrondie au ×5 supérieur

Karpathy Guidelines
.claude/skills/karpathy-guidelines.md · chargé avec tout skill domaine
Principe 1
Réfléchir avant d'agir
Avant toute modification :
• Lire le bloc %% concerné et les 2 blocs adjacents
• Vérifier patterns/ : pattern applicable ?
• Énoncer ce qui va être modifié
• Attendre la confirmation
Principe 2
Simplicité stricte
• Minimum de code qui satisfait correction-commune + skill domaine
• Zéro \textbf{} décoratif non prévu
• Zéro phrase superflue
• Étape > 3 lignes pour un calcul simple → réécrire
Principe 3
Modifications chirurgicales
• Seul le bloc %% SOLUTION demandé est modifié
• Blocs %% voisins : intouchables
• Commentaires % durée estimée : jamais touchés
• Test : diff = uniquement les lignes du bloc
Principe 4
Checklist obligatoire
Avant chaque insertion, afficher :
✓ Bloc ciblé · Pattern appliqué
✓ Style (ACTIVITE / APPLICATION)
✓ Phrase ▶ présente/absente
✓ Conclusion présente/absente
✓ Blocs voisins non modifiés
🎨
Styles de correction
Distinction fondamentale ACTIVITE vs APPLICATION
SOLUTION_ACTIVITE — Découverte
Pont explicite avec acquis du collège
Phrase ▶ \blacktriangleright OBLIGATOIRE après chaque notion
Vocabulaire accessible, début lycée
Ne jamais écrire "Conclusion"
Connecteurs \Leftrightarrow si pertinents
Institutionnalisation avec \textcolor{PrimaryColor}{...}
SOLUTION_APPLICATION — Mobilisation
Connecteurs logiques obligatoires : On a → \Leftrightarrow → Donc
Phrase ▶ \blacktriangleright INTERDITE
Justifications à chaque étape
Conclusion standard : Donc S = {...}.
Style du skill domaine appliqué
Rigueur de modèle

Justifications standard (correction-commune)

OpérationJustification
Développer(distributivité)
Regrouper(on regroupe les termes semblables)
Diviser par c > 0(on divise par c, c > 0)
Diviser par c < 0(on divise par c < 0, le sens de l'inégalité s'inverse)
Multiplier par PPCM(on multiplie par n, PPCM de a et b)
Factoriser(factorisation)

Conclusions standard

TypeFormat
ÉquationDonc $S = \{valeur\}$.
InéquationDonc $S = [a \; ; \; +\infty[$.
Pas de solutionDonc $S = \emptyset$.
Tout réelDonc $S = \R$.
